## 3. Logical vs Physical Address

| Address | Meaning |
|---|---|
| Logical address | address generated by CPU; also called virtual address |
| Physical address | actual address sent to memory unit |

Important:

- User program deals with logical addresses.
- Memory hardware uses physical addresses.
- Runtime mapping is done by MMU.

MCQ trap:

- Logical and physical addresses are same in compile-time/load-time binding, but different in execution-time binding.

## 4. Address Binding

Address binding maps program addresses to memory addresses.

| Binding time | Meaning | Logical = Physical? |
|---|---|---|
| Compile time | memory location known before compilation | yes |
| Load time | final address decided when program is loaded | yes after loading |
| Execution time | address translation during execution | no |

Execution-time binding:

- Requires hardware support.
- Uses MMU.
- Allows process movement during execution.

MCQ clue:

- If process can move during execution, binding must be execution-time.

## 5. Memory Management Unit (MMU)

MMU is hardware that maps logical addresses to physical addresses at runtime.

### Base and Limit Registers

| Register | Meaning |
|---|---|
| Base / relocation register | smallest physical address of process |
| Limit register | size/range of legal logical address space |

Address translation:

```text
if logical_address < limit:
    physical_address = base + logical_address
else:
    trap to OS
```

Example:

```text
base = 14000
limit = 1000
logical address = 346
physical address = 14000 + 346 = 14346
```

If logical address is 1200:

```text
1200 >= 1000, so trap
```

GATE trap:

- Valid condition is usually `logical address < limit`, not `<=`.

## 6. Swapping

Swapping temporarily moves a process from main memory to backing store and later brings it back.

Backing store:

- Usually disk.
- Must be large and fast enough.

Roll out, roll in:

- Used with priority scheduling.
- Lower-priority process swapped out.
- Higher-priority process loaded.

MCQ trap:

- Traditional swapping moves entire process.
- Modern systems often page individual memory pages instead.

## 7. Contiguous Memory Allocation

In contiguous allocation, each process gets one continuous block of memory.

Memory is often divided into:

- OS part.
- User process part.

### Fixed Partitioning

Memory divided into fixed-size partitions.

Problem:

- Internal fragmentation.

### Variable Partitioning

Partitions are created dynamically based on process size.

Problem:

- External fragmentation.

## 8. Dynamic Storage Allocation

Given free holes, choose where to place process.

| Strategy | Rule | Advantage | Disadvantage |
|---|---|---|---|
| First fit | first hole big enough | usually fastest | may leave small holes near beginning |
| Best fit | smallest hole big enough | leaves least leftover for that allocation | creates many tiny holes |
| Worst fit | largest hole | leaves large leftover hole | often poor utilization |

GATE trick:

- Do not assume best fit is always best overall.
- Best fit minimizes leftover for current allocation, not necessarily future fragmentation.

## 9. Fragmentation

### Internal Fragmentation

Allocated memory is larger than requested memory.

Example:

```text
Block size = 100 KB
Process needs = 92 KB
Internal fragmentation = 8 KB
```

Common in:

- Fixed partitions.
- Paging.

### External Fragmentation

Total free memory is enough, but it is not contiguous.

Example:

```text
Free holes: 10 KB + 20 KB + 30 KB = 60 KB total
Request: 50 KB
Cannot allocate if no single 50 KB hole exists
```

Common in:

- Variable partitions.
- Segmentation.

### Compaction

Compaction moves memory contents to combine scattered holes.

Requires:

- Dynamic relocation/execution-time binding.

MCQ trap:

- Paging eliminates external fragmentation, not internal fragmentation.
- Segmentation can suffer external fragmentation.

## 10. Paging

Paging allows non-contiguous physical memory allocation.

Basic idea:

| Logical memory | Physical memory |
|---|---|
| divided into pages | divided into frames |

Important:

- Page size = frame size.
- Page number indexes page table.
- Page table gives frame number.
- Offset stays unchanged.

### Address Format

Logical address:

```text
page number p + page offset d
```

Physical address:

```text
frame number f + frame offset d
```

If page size is power of 2:

```text
number of offset bits = log2(page size)
number of page bits = logical address bits - offset bits
```

### Address Translation Formula

```text
page_number = logical_address / page_size
offset = logical_address % page_size
frame_number = page_table[page_number]
physical_address = frame_number * page_size + offset
```

Example:

```text
Logical address = 2500
Page size = 1000
Page number = 2500 / 1000 = 2
Offset = 2500 % 1000 = 500
If page 2 maps to frame 7:
Physical address = 7 * 1000 + 500 = 7500
```

GATE bit trick:

If page size = 4 KB:

```text
4 KB = 2^12 bytes
offset bits = 12
```

If logical address space = 32 bits:

```text
page number bits = 32 - 12 = 20
number of pages = 2^20
```

## 11. Page Table

Page table maps page numbers to frame numbers.

Each process has its own page table.

Page Table Base Register (PTBR):

- Points to page table location in memory.

Page Table Length Register (PTLR):

- Stores size of page table.

Problem:

- If page table is in memory, each data access may require two memory accesses:
  - one for page table entry
  - one for actual data

Solution:

- TLB.

## 12. Translation Lookaside Buffer (TLB)

TLB is a small, fast associative cache of recent page-table entries.

| Event | Meaning |
|---|---|
| TLB hit | page entry found in TLB |
| TLB miss | page table must be checked |
| Hit ratio | fraction/percentage of references found in TLB |

### TLB EAT Formula

Let:

```text
h = hit ratio
t = memory access time
epsilon = TLB lookup time
```

Then:

```text
EAT = h * (epsilon + t) + (1 - h) * (epsilon + 2t)
```

If TLB lookup time is ignored:

```text
EAT = h * t + (1 - h) * 2t
```

MCQ trap:

- TLB miss means two memory accesses in simple paging: one page-table access and one actual memory access.
- TLB hit means one actual memory access plus TLB lookup.

## 13. Multi-Level Paging

Used when page table is too large.

Idea:

- Page table itself is paged.
- Logical address split into outer page number, inner page number, and offset.

MCQ clue:

- Large logical address space often uses two-level or multi-level paging.

## 14. Segmentation

Segmentation matches user's view of memory.

Program is divided into logical segments:

- main program
- procedures/functions
- stack
- arrays
- symbol table

Logical address:

```text
segment number + offset
```

Segment table entry:

```text
base + limit
```

Translation:

```text
if offset < limit:
    physical address = base + offset
else:
    trap
```

Advantages:

- Matches programmer view.
- Easy sharing/protection at segment level.

Disadvantages:

- External fragmentation.

## 15. Paging vs Segmentation

| Paging | Segmentation |
|---|---|
| fixed-size pages | variable-size segments |
| OS/hardware view | user/programmer view |
| eliminates external fragmentation | may cause external fragmentation |
| may cause internal fragmentation | little/no internal fragmentation inside exact segment |
| logical address: page + offset | logical address: segment + offset |

MCQ trap:

- Page size is fixed.
- Segment size is variable.

## 16. Segmentation with Paging

Combines advantages:

- Segmentation gives logical view.
- Paging reduces external fragmentation.

Idea:

```text
segment number -> segment table -> page table for that segment
segment offset -> page number + page offset
```

Useful because:

- Each segment can be paged separately.

## 17. Virtual Memory

Virtual memory separates logical address space from physical memory.

It allows:

- execution of programs larger than physical memory.
- more processes in memory.
- less initial loading.
- better CPU utilization.

Common implementation:

- Demand paging.

MCQ trap:

- Virtual memory does not mean infinite memory.
- It uses disk/backing store to extend apparent memory.

## 18. Demand Paging

Demand paging brings a page into memory only when it is needed.

Instead of loading entire process:

```text
Load pages on demand
```

Page table has valid-invalid bit:

| Bit | Meaning |
|---|---|
| valid | page is in memory |
| invalid | page is not in memory or reference invalid |

## 19. Page Fault

A page fault occurs when a process references a page not currently in physical memory.

Page fault handling:

1. Trap to OS.
2. Check whether reference is valid.
3. If invalid, terminate process.
4. If valid but not in memory, locate page on disk.
5. Find a free frame.
6. Read page into frame.
7. Update page table and valid bit.
8. Restart interrupted instruction.

MCQ trap:

- Page fault is not always an error. It is normal in demand paging if the reference is valid but page is absent.

## 20. Page Fault Effective Access Time

Formula:

```text
EAT = (1 - p) * memory_access_time + p * page_fault_service_time
```

where:

```text
p = page fault rate
```

More complete version may include normal memory access in fault path, but exam options usually use simplified formula.

Important unit conversion:

```text
1 ms = 10^6 ns
1 microsecond = 10^3 ns
```

Example:

```text
memory access = 20 ns
page fault service = 10 ms = 10,000,000 ns
page fault rate = 1 / 1,000,000 = 0.000001

EAT = (0.999999 * 20) + (0.000001 * 10,000,000)
    approximately 20 + 10
    = 30 ns
```

GATE trick:

- Even a tiny page fault rate can dominate EAT because disk service time is huge.

## 21. Page Replacement

If no free frame is available during page fault, OS must choose a victim page to remove.

Goal:

- Minimize page fault rate.

If victim page is modified:

- write it back to disk.

If not modified:

- simply overwrite.

Dirty/modify bit:

- indicates whether page has been changed.

## 22. FIFO Page Replacement

Rule:

- Replace the page that entered memory earliest.

Advantages:

- Simple.

Disadvantages:

- May remove heavily used page.
- Suffers from Belady's anomaly.

Belady's anomaly:

- More frames can lead to more page faults.

MCQ trap:

- FIFO is not always improved by increasing frames.

## 23. Optimal Page Replacement

Rule:

- Replace page that will not be used for longest time in future.

Advantages:

- Minimum possible page faults.
- Used as benchmark.

Disadvantages:

- Not implementable exactly because future references are unknown.

MCQ clue:

- "Looks into future" means Optimal.

## 24. LRU Page Replacement

LRU = Least Recently Used.

Rule:

- Replace page that has not been used for longest time in past.

Advantages:

- Good practical approximation of Optimal.
- Does not suffer from Belady's anomaly.

Implementation:

- Counters/timestamps.
- Stack.

MCQ clue:

- "Looks into past" means LRU.

## 25. Page Replacement Tracing Method

For each reference:

1. If page already in frames: hit.
2. If page not in frames: page fault.
3. If free frame exists: insert page.
4. Else choose victim according to algorithm.
5. Update order/history.

For FIFO:

- Maintain queue of arrival order.

For LRU:

- Maintain recent-use order.

For Optimal:

- Look ahead in reference string.
- Replace page used farthest in future.
- If never used again, replace it immediately.

## 26. Classic Page Replacement Example

Reference string:

```text
7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2
```

Frames:

```text
3
```

Expected behavior:

- FIFO is easy but may do extra faults.
- Optimal should give lowest faults.
- LRU is usually close to Optimal.

Exam tip:

- Do not mentally guess. Draw frame table.
- Mark hits and faults separately.

## 27. Allocation of Frames

If there are `m` frames and `n` processes, OS must decide frame distribution.

| Method | Meaning |
|---|---|
| Equal allocation | each process gets same number of frames |
| Proportional allocation | frames based on process size |
| Priority allocation | frames based on process priority |

Minimum frames:

- Each process must get enough frames to execute the most complex instruction that may reference memory.

## 28. Global vs Local Replacement

| Type | Meaning | Effect |
|---|---|---|
| Global replacement | process can select victim from all frames | higher throughput, more variable performance |
| Local replacement | process selects victim only from its own frames | more predictable, may underuse memory |

MCQ trap:

- Global replacement allows one process to take another process's frame.
- Local replacement restricts victim selection to same process.

## 29. Thrashing

Thrashing occurs when a process spends more time paging than executing.

Cause:

- Not enough frames for active working set.
- Too high degree of multiprogramming.

Symptoms:

- High page fault rate.
- Low CPU utilization.
- Excessive disk I/O.

Bad feedback loop:

```text
CPU utilization falls
OS adds more processes
more memory pressure
more page faults
CPU utilization falls further
```

## 30. Locality of Reference

Programs tend to access a small set of pages repeatedly for some time.

Types:

| Type | Meaning |
|---|---|
| Temporal locality | recently used item likely used again |
| Spatial locality | nearby addresses likely used soon |

Virtual memory depends heavily on locality.

## 31. Working-Set Model

Working set:

- Set of pages referenced in the most recent fixed window of references.

Working-set window:

```text
Delta
```

Demand:

```text
D = sum of working-set sizes of all processes
```

If:

```text
D > total available frames
```

Then:

- thrashing may occur.
- OS should suspend/swap out a process.

## 32. Page Fault Frequency

Another thrashing control method.

Idea:

- If page fault rate is too high, give process more frames.
- If page fault rate is too low, take frames away.

## 33. High-Yield Differences

### Internal vs External Fragmentation

| Internal | External |
|---|---|
| waste inside allocated block | waste between allocated blocks |
| fixed-size allocation | variable-size allocation |
| paging can cause it | segmentation can cause it |

### Paging vs Virtual Memory

| Paging | Virtual memory |
|---|---|
| memory-management scheme | abstraction that allows large logical memory |
| divides memory into pages/frames | often implemented using demand paging |
| can exist without demand paging | demand paging is virtual memory technique |

### Page Fault vs TLB Miss

| TLB miss | Page fault |
|---|---|
| page-table entry not found in TLB | page not present in physical memory |
| handled by checking page table | requires OS trap and disk I/O |
| relatively fast | very slow |

MCQ trap:

- TLB miss does not always mean page fault.
- Page fault means page is absent from RAM.

## 34. GATE-Level Problem Types

### Type 1: Address Translation

Given:

- logical address
- page size
- page table

Find:

- page number
- offset
- frame number
- physical address

Use:

```text
p = LA / page_size
d = LA % page_size
PA = frame * page_size + d
```

### Type 2: Number of Pages/Page Table Entries

Given logical address space and page size:

```text
number of pages = logical address space / page size
```

If address bits are given:

```text
number of pages = 2^(logical address bits - offset bits)
```

### Type 3: TLB EAT

Use hit ratio carefully.

If hit ratio is 90%, use:

```text
h = 0.9
```

### Type 4: Page Fault EAT

Convert units first.

If service time is ms and memory time is ns:

```text
convert ms to ns
```

### Type 5: Page Replacement Fault Count

Always draw frame table.

Use:

- FIFO queue.
- LRU recent-use order.
- Optimal future lookahead.

### Type 6: Fragmentation Identification

Question says:

```text
enough total memory but not contiguous
```

Answer:

- external fragmentation.

Question says:

```text
allocated block bigger than requested
```

Answer:

- internal fragmentation.

## 35. Common Mistakes

1. Confusing page and frame.
2. Saying paging causes external fragmentation.
3. Forgetting offset stays same in paging.
4. Using `<= limit` instead of `< limit` in base-limit checks.
5. Treating TLB miss as page fault.
6. Forgetting to convert ms to ns in EAT.
7. Replacing wrong page in LRU by confusing least recent with oldest loaded.
8. Saying Optimal is practically implementable.
9. Saying more frames always reduce page faults; FIFO can violate this.
10. Confusing global and local replacement.
